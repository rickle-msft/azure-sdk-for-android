TokenRequestHandle
Link copied to clipboard

A handle that a background running thread (e.g. OkHttp thread) uses to synchronously receive an AccessToken produced by the main UI thread. A background running thread can initiate the AccessToken request by calling () } , which returns a TokenRequestHandle instance. The background thread then waits for the arrival of the AccessToken by calling waitForToken .

IMPLEMENTATION NOTE: The TokenRequestObservable INTERNALLY uses the same TokenRequestHandle object (returned from ) as an Event to notify a TokenRequestObserver about the need for AccessToken . Upon receiving this event, the TokenRequestObserver invokes onTokenRequest ( TokenResponseCallback ) } which is responsible for retrieving the AccessToken , the observer then sets the AccessToken or AccessToken retrieval error in the Handle using it's package PRIVATE setters setToken , setError .

It is important that an Event instance is processed only once by the TokenRequestObserver . The TokenRequestObservable uses a androidx.lifecycle.LiveData instance to post the Event to TokenRequestObserver . androidx.lifecycle.LiveData is designed to cache the latest value (in this case Event) and re-deliver it when there is a new Observer. An example for unexpected re-delivery is: Let's say the TokenRequestObservable is owned by a androidx.lifecycle.ViewModel which lives through multiple {@code Activity::onCreate} - {@code Activity::onDestroy} spans and in {@code Activity::onCreate} we register a TokenRequestObserver to this androidx.lifecycle.ViewModel 's scoped Observable. When a background thread requests for a token, the Observer will receive the request and process it. If the user begins rotating the screen (i.e configuration changed) then {@code Activity::onDestroy} is called and once the rotation is done {@code Activity::onCreate} method will be called again, which will result in the registration of a new TokenRequestObserver and will internally cause androidx.lifecycle.LiveData to re-deliver the old cached/handled Event, which we definitely don't want the new Observer to handle. To identify such an Event, the Observer uses isConsumed .

TokenRequestObservable
Link copied to clipboard

An Observable to send the token request from a background thread to an Observer in the main UI thread.

A use case of this type is - sending a token request from androidx.lifecycle.ViewModel code running in a background thread to the main UI thread, where the actual acquisition of token is interactive, e.g. it involves android.app.Activity UI. This Observable decouples the androidx.lifecycle.ViewModel from the UI so that the former will not hold a reference to the UI component, holding such reference can lead to a memory leak. However, the Observer of this Observable can hold a UI reference, because the Observer is removed when the UI that the Observer is associated with is in the DESTROYED state.
